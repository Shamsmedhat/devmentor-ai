DevMentor AI — Project Walkthrough A guided tour of every layer, file, and data flow before you start the RAG feature.

TL;DR — 60-second tour

DevMentor AI is an Arabic-first (with English) Next.js 16 chat app where a developer talks to an AI mentor that streams Markdown/code answers. A request enters through src/proxy.ts (next-intl + Supabase auth), lands on a localized App-Router page, the chat UI mounts client components that delegate to two custom hooks (useChat wrapping the Vercel AI SDK, useChatSession managing the session row), and submitting a message hits POST /api/chat, which authenticates → rate-limits → strips unsupported attachments → streams from Groq Llama 3.3 70B back through the UIMessage protocol. Sessions and messages are persisted in Supabase Postgres; the sidebar subscribes to a realtime channel to update the list as new sessions appear.

What's running

  • Frontend: Next.js 16 App Router, React 19, Tailwind v4 + shadcn/ui
  • i18n: next-intl, locales ar (default) + en, prefix always shown
  • Auth + DB: Supabase (Google OAuth, Postgres, Realtime, RLS)
  • AI: Vercel AI SDK 6 + @ai-sdk/groq → llama-3.3-70b-versatile
  • Streaming: UIMessage stream protocol (toUIMessageStreamResponse ↔ useChat)

What you'll touch for RAG

  • src/app/api/chat/route.ts — inject retrieval before streamText
  • src/lib/ai/prompts.ts — already has RAG_SYSTEM_PROMPT(context)
  • New: src/lib/ai/rag.ts, embedding service, ingest action/route
  • Supabase: enable pgvector, add documents + chunks tables

Flow 1 — App boot & locale resolution

  1. Browser requests /ar/anything (or /en/anything).
  2. src/proxy.ts runs. next-intl middleware sets the locale cookie and may rewrite the URL; localePrefix: "always" means the prefix is required.
  3. The same proxy then builds a server Supabase client with cookie helpers and calls supabase.auth.getUser() — this both reads the session and refreshes its cookies onto the response.
  4. Path is split into segments; root = first non-locale segment. If it is chat or review and there's no user → 302 /[locale]/login?next=…. If root is login and there is a user → 302 safeNextPath().
  5. Otherwise the response continues into Next.js routing. Root app/layout.tsx (just CSS) → app/[locale]/layout.tsx resolves messages, fonts, html dir/lang, and renders NextIntlClientProvider.
  6. The matching route layout ((auth) or (dashboard)) wraps the page, then the page renders.
Why proxy.ts not middleware.ts? Next.js 16 renamed the file. Same purpose, new name. Your repo's AGENTS.md warns about exactly this kind of drift from older docs.

Flow 2 — Google OAuth login

  1. User on /[locale]/login clicks the Google button. GoogleLoginButton (client) calls useGoogleLogin({ nextPath }).
  2. signInWithGoogle() wraps the call in useTransition and asks Supabase for an OAuth URL. The redirectTo is computed as <origin>/api/auth/callback?next=<safe>.
  3. Browser leaves to Google, user consents, Google sends them back to /api/auth/callback?code=…&next=….
  4. The callback route reads code and next, derives locale via localeFromNextParam, sanitizes via safeNextPath, builds a server Supabase client, calls exchangeCodeForSession(code).
  5. On success → 302 to the cleaned next. On error → 302 to /[locale]/login?error=auth_failed&next=… so the page shows a banner.
  6. Now any subsequent request goes through proxy.ts, which sees a user cookie and lets /chat through.
Open-redirect guard: safeNextPath rejects anything that isn't /<locale>/..., contains //, or has a protocol. Always go through it before redirecting on user-supplied next.

Flow 3 — Sending a chat message (the hot path)

  1. User types in ChatInput → state lives in useChat (input) and attachments in attachments[].
  2. Pressing Enter (or Send) calls ChatSessionPanel.handleSend. It first ensures a session exists: if sessionId is null, it calls useChatSession.createSession, which invokes createChatSessionAction (server). That INSERTs into chat_sessions and returns the new id.
  3. The user message is persisted via saveChatMessageAction({ role: "user", content }). Content is flattened by formatUserMessageForPersistence so attachments are stored as readable references.
  4. If this is the first message, updateSessionTitleAction is called with the first 40 characters as the title — this is what shows in the sidebar list.
  5. useChat.handleSubmit fires. DefaultChatTransport.prepareSendMessagesRequest appends { sessionId, attachments } to the body, then POST /api/chat goes out with the full UIMessage history.
  6. On the server, route.ts:
    • builds a server Supabase client, calls getUser(); missing user → 401.
    • checkChatRateLimit(userId) HEAD-counts user messages in the last hour against AI_LIMITS.CHAT_MESSAGES_PER_HOUR (50). Over → 429 with reset.
    • normalizeUiMessagesForAttachmentCapabilities walks each file part. The active model (groq-llama-3.3-70b) flags both image and non-image as unsupported, so file parts get dropped and replaced with a textual marker — Groq Llama never sees binary attachments.
    • convertToModelMessages turns UIMessages into model messages.
    • streamText({ model: groq("llama-3.3-70b-versatile"), system: MENTOR_SYSTEM_PROMPT, messages }) opens a Groq stream.
    • result.toUIMessageStreamResponse() sends the stream back in the UIMessage protocol.
  7. useChat on the browser consumes the stream. As tokens arrive, messages updates and the assistant bubble re-renders incrementally. Code blocks render through react-markdown + Prism (oneDark).
  8. When the stream finishes, onFinish(message) fires inside ChatSessionPanel: it persists the assistant turn via saveChatMessageAction({ role: "assistant", content }) and notifies the parent so the sidebar can refresh ordering.
  9. If the user scrolled up, autoscroll is suppressed (distFromBottom < 120 check in handleScroll); otherwise the view sticks to the bottom via scrollIntoView.
Key invariant: the database is the source of truth for history. useChat holds the live stream state, but reload = fetch chat_messages via getChatMessagesAction, then bump panelKey to remount ChatSessionPanel with fresh initialMessages.

Flow 4 — Loading an existing chat

  1. User clicks a session in the sidebar → onSessionSelect(sessionId) bubbles up to ChatShell, which sets selectedSessionId.
  2. ChatArea sees the change in its useEffect, calls getChatMessagesAction({ userId, sessionId }), and stores the result in loadedMessages.
  3. It bumps panelKey — React unmounts the previous ChatSessionPanel and mounts a new one with initialMessages={loadedMessages}, which seeds useChat.
  4. Two refs (syncingFromCreateRef, skipFirstNullClearRef) guard against double-fetches and an unwanted clear when a brand-new session id is created locally.

Flow 5 — Sidebar realtime updates

  1. On mount, Sidebar queries chat_sessions directly with the browser Supabase client, ordered by updated_at DESC.
  2. It then opens a Supabase Realtime channel listening for postgres_changes on chat_sessions, filtered by user_id = auth.uid().
  3. On INSERT/UPDATE/DELETE events, the local sessions array is patched — so a new chat created in another tab appears live.
  4. Sign-out: supabase.auth.signOut() → router.push("/"). The proxy will redirect any subsequent protected access back to login.

Key data shapes (one place to look)

UIMessage (Vercel AI SDK 6)

type UIMessage = {
  id: string;
  role: "user" | "assistant" | "system";
  parts: Array<
    | { type: "text"; text: string }
    | { type: "file"; mediaType: string; data: string /* base64 */ ; name?: string }
    /* + tool, reasoning, etc. */
  >;
};

This is what flows over POST /api/chat and what useChat stores. convertToModelMessages turns it into provider-specific messages before streamText.

Persisted message content

chat_messages.content is a single text column. formatUserMessageForPersistence flattens text + attachment metadata into that string when saving the user turn.

Things to decide before writing code

  • Embedding provider (Groq doesn't expose embeddings). OpenAI text-embedding-3-small is cheap and good enough; Voyage is stronger; Cohere has multilingual.
  • Chunking strategy — by tokens (recommended ~500 with 50 overlap), by markdown sections, by code symbols, etc.
  • Index/search — pgvector + cosine similarity is the simplest path; HNSW index for performance.
  • Retrieval surface — automatic on every chat? Tied to a "knowledge base" picker? Per-session or global?
  • RLS — chunks should be readable only by the document owner unless explicitly shared.

Cheat sheet

"Where do I add…?"

  • A new locale → src/i18n/routing.ts + messages/<locale>.json
  • A protected page → put it under /[locale]/(dashboard)/ and add the segment name to the protected check in proxy.ts
  • A new model → extend model-capabilities.ts + flip CHAT_MODEL_CAPABILITY_KEY
  • A new server mutation → action in src/lib/actions/*.action.ts, query in src/lib/services/*.service.ts

"How is auth enforced?"

  • Pages: proxy.ts redirects unauthed
  • API: route.ts calls getUser(), returns 401
  • Actions: verify session.user_id === userId
  • DB: RLS — last line of defense

"Why is X happening?"

  • Sidebar updates without refresh → postgres_changes Realtime channel.
  • First message sets the title → updateSessionTitleAction with first 40 chars.
  • File upload silently dropped → model capability says no, normalize step replaces with marker.
  • Locale didn't switch → check localePrefix: "always" + switchLocale in chat-area.tsx.
